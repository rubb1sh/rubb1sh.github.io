HTTP协议

发表于 2019-08-03   |   分类于 计算机网络

HTTP特性

  1. http协议是构建在tcp/ip协议之上的，一个应用层协议，默认端口号为80、8080
  2. http是无连接无状态的

HTTP报文

1. 请求报文

HTTP协议是以ASCII码传输的，建立在tcp/ip协议上的应用层规范。规范把HTTP请求分为三个部分：请求行(request-line)、请求头(header)、请求体(request-body)

HTTP定义了与服务器交互的不同方法，最基本的方法包括4种。分别是GET、POST、PUT、DELETE。

  1. GET用于信息获取，而且应该是幂等和安全的 （GET得幂等，在开发中要记住这个规范）
  2. POST一般用于修改服务器上的资源的请求
  3. HEAD查看请求http的头部信息
  4. PUT支持幂等的POST
  5. DELETE删除资源
  6. OPTION查看支持的方法

    GET和POST的区别：

    实际没有什么区别，只是GET请求内容在url上，而POST在请求体（request body）。

POST请求提交数据

HTTP请求post提交数据必须在请求体中，但是没有规定解析请求的编码格式，因此服务端通常根据请求头（header）中的Content-Type字段来获知。

响应报文结构

  • 状态行
  • 相应头
  • 响应正文

常见响应码：

  • 200 ok 客户端请求成功
  • 301 Moved Permanently 请求永久重定向（如果使用301，浏览器应该该请求已经被永久转移，下次访问这个url时，应该直接访问重定向的url，不该访问这个url）
  • 302 Moved Temporarily 请求暂时重定向（302的不会像301）
  • 304 Not Modified 文件未修改，可以使用缓存文件
  • 400 Bad Request 客户端请求语法有错误，不能被服务器解析
  • 401 unauthorized 用于用户验证
  • 403 Forbidden 服务器拒绝响应，一般是被发现是爬虫时返回
  • 404 Not Found 请求的资源不存在
  • 500 Internal Server error 服务器错误
  • 501 Not Implemented 客户端发起的请求超出服务器的能力范围（比如，使用了服务器不支持的请求方法）时，使用此状态码
  • 502 Bad Gateway 作为代理或网关使用的服务器从请求响应链的下一条链路上收到了一条伪响应（比如，它无法连接到其父网关）时，使用此状态码
  • 503 Service Unavailable 服务器当前不能处理该请求
  • 504 Gateway Timeout 服务器作为网关或代理，但是没有及时从上游服务器收到请求
  • 505 HTTP Version Not Supported http协议版本不支持

持久连接

HTTP 1.1中，有keep-Alive规范

  • HTTP Keep-Alive 简单说就是保持当前的TCP连接，避免了重新建立连接。
  • HTTP 长连接不可能一直保持，例如 Keep-Alive: timeout=5, max=100，表示这个TCP通道可以保持5秒，max=100，表示这个长连接最多接收100次请求就断开。

    Transfer-Encoding

      Transfer-Encoding 是一个用来标示 HTTP 报文传输格式的头部值。尽管这个取值理论上可以有很多，但是当前的 HTTP 规范里实际上只定义了一种传输取值——chunked。如果一个HTTP消息（请求消息或应答消息）的Transfer-Encoding消息头的值为chunked，那么，消息体由数量未定的块组成，并以最后一个大小为0的块为结束。

  multipart 是一种 Content-Type，标示 HTTP 报文内容的类型，而 chunked 是一种传输格式，标示报头将以何种方式进行传输

会话跟踪

概念：会话跟踪指的是对同一个用户对服务器的连续的请求和接受响应的监视。

为什么需要会话跟踪：
  浏览器与服务器之间的通信是通过HTTP协议进行通信的，而HTTP协议是”无状态”的协议，它不能保存客户的信息，即一次响应完成之后连接就断开了，下一次的请求需要重新连接，这样就需要判断是否是同一个用户，所以才有会话跟踪技术来实现这种要求。
会话跟踪的方法：
i. URL重写
ii. 隐藏表单域
iii. Cookie 服务器端发送一段信息给客户端，用于标识一个用户。Cookie分两种，一种是放在内存中，一种是放在硬盘中。且cookie是默认禁用的。
iv. Seesion 服务器端创建一个session对象，并和一个sessionID绑定，把sessionid发给用户，用户发送请求时会一起发送这个sessionid，进而服务器端就可以识别不同的用户。session是基于cookie的，cookie被禁用，session也将失效。
